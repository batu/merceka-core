# Problem

`merceka_core/merceka_core/llm.py` is an AUTOGENERATED file — its header
even announces it. But over several months, three commits landed that
edited the `.py` directly without back-porting to `nbs/00_llm.ipynb`:

- `99c4b1a` — Claude CLI provider + async paths + local Ollama vision
- `a07c9af` — native tool calling via `--allowedTools` and `--add-dir`
- `33e0826` — token-by-token streaming via stream-json

By the time I picked up merceka-phase-0, the `.py` was 975 lines and
the notebook was ~680 lines. Running `nbdev_export` on the stale
notebook would have **silently deleted 290 lines of working code** —
Claude CLI support, streaming, the fallback cascade, `agenerate_with_resource`,
`stream_generate`, and more.

No tooling caught this:
- `nbdev_clean` only strips outputs; doesn't flag missing-code drift.
- CI doesn't run `nbdev_export` and diff against committed `.py`.
- The project memory `feedback_prefer_merceka_core.md` says "extend
  merceka, don't bypass it" but doesn't address the notebook-vs-.py
  invariant directly.

The drift is silent because:
- `.py` changes pass typecheck, pass tests, pass review.
- No one opens the notebook during a normal PR workflow.
- The `nbdev_export` regression only surfaces when SOMEONE ELSE later
  tries to edit the notebook per policy.

# Fix this session

Before making any new notebook edits, I:
1. Copied `llm.py` → `/tmp/llm_before.py`.
2. Ran `nbdev_export` to see what the notebook-of-record would produce.
3. Diffed. Confirmed 290 lines were being dropped.
4. Wrote `/tmp/sync_nb_from_py.py` — a one-shot script that extracts
   each stale cell's target content from the current `.py` and
   overwrites the notebook cell's source. Three cells needed it:
   cell 2 (imports + CLAUDE_CLI_TIMEOUT + _logger), cell 15 (added
   `create_ollama_vision_message`), cell 31 (the big LLM class).
5. Ran `nbdev_export` again and confirmed byte-level parity (modulo
   the header comment change from "nbsync" to the standard nbdev
   header).
6. THEN made the phase-0 additions on top.

# Rule to install

**Before editing an nbdev notebook, round-trip `nbdev_export` and
diff.** If there's a semantic diff between the freshly regenerated
`.py` and the committed `.py`, the notebook is stale. Do NOT proceed
with edits until you back-port the stale cells.

**Before editing an nbdev `.py` directly, STOP.** The `.py` header
says AUTOGENERATED for a reason. Route through the notebook.

## Optional CI check

Add a pre-push or CI step that fails when regenerated `.py` differs
from committed `.py`:

```bash
uv run nbdev_export
git diff --exit-code merceka_core/
```

This would have caught the three drift commits as they landed. File
as a follow-up card if the merceka team wants to enforce the
invariant mechanically.

# Why it matters for slab

Slab has many sibling repos that will also need to grow. Every one of
them becomes a drift-risk if they adopt nbdev. The alternative — pure
.py with plain pytest — is what videototext / quest / slab are going
to use per the deepened plan. Merceka is the only nbdev holdout, and
even there, `errors.py` / `resources/gpu.py` in this card are plain
modules precisely because they're infra, not LLM-behavior literate
docs. Keep nbdev scope narrow.
