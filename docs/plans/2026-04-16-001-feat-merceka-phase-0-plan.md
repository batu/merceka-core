# merceka-phase-0 — OpenRouter Claude vision + Gemini video + gpu_lock

## Scope (ground truth from card)

Three new `merceka_core.LLM` capabilities plus a small exception hierarchy and a cross-process GPU file lock, with a same-PR mindweaver migration from its in-process `asyncio.Lock` to the new file lock. No changes to slab/videotoframes/videototext/quest.

Everything flows through the nbdev pipeline: edit `nbs/00_llm.ipynb`, run `nbdev_export` to regenerate `merceka_core/llm.py`. Non-LLM modules (`resources/gpu.py`, `errors.py`) are plain Python under `merceka_core/` because they are new surfaces, not derivations of existing notebook cells.

## Decisions this plan records

Each decision is a reversible micro-choice that the card description leaves to the implementer. Recorded here so /ce:review can audit them against intent rather than guessing.

### D1 — `nbdev_export`, not `nbsync update`
The card description references `nbsync update`. This repo is pure nbdev (no `nbsync` binary is installed in `.venv/bin/` or in `uv tool list`). The regeneration command is `nbdev_export` (or `uv run nbdev_export` from the repo root). The `merceka_core/llm.py` header comment says "AUTOGENERATED by nbsync" but that is stale naming — the mechanism is nbdev. No action beyond using `nbdev_export` and, if editing the header naming surfaces as a /ce:review point, leave it alone (auto-regen on next card fixes it).

### D2 — `errors.py` and `resources/gpu.py` are plain `.py`, not notebook cells
The card description places these modules outside `merceka_core/llm.py`. They are new surfaces with no prior notebook heritage, and the `literate-notebook` skill explicitly covers LLM prompt workbenches — not infra primitives. Putting them as plain modules keeps the notebook focused on LLM behavior and lets the file-lock test suite execute without kernel startup overhead.

`merceka_core/__init__.py` re-exports `gpu_lock`, `GpuLockTimeout`, `VideoUploadError`, `VideoBackendError`, `VideoNotFoundError` so downstream imports read `from merceka_core import gpu_lock` rather than reaching into private submodules.

### D3 — Gemini SDK: `google-genai>=1.0`
Add as a runtime dep in `pyproject.toml`. User approval for new deps is pre-authorized inside the trello-pipeline skill (the card description names the SDK explicitly). The legacy `google-generativeai` package is NOT added.

Auth: `google-genai` uses `GOOGLE_API_KEY` env var by default (same as `google-generativeai`'s `genai.configure`). The client constructor also accepts `api_key=...`. We use the env-var path — matches how OpenRouter auth works in `llm.py` (`os.getenv("OPENROUTER_API_KEY")`).

Block in a thread so the event loop stays responsive (async version uses `asyncio.to_thread`). After `ACTIVE`, pass the `File` handle to `generate_content`.

### D6 — `gpu_lock` primitive
```python
# merceka_core/resources/gpu.py
import asyncio, contextlib, fcntl, os
from pathlib import Path

from merceka_core.errors import GpuLockTimeout

_LOCK_PATH = Path.home() / ".local" / "state" / "utolye" / "gpu.lock"

@contextlib.asynccontextmanager
async def gpu_lock(timeout: float | None = None):
    _LOCK_PATH.parent.mkdir(parents=True, exist_ok=True)
    fd = os.open(_LOCK_PATH, os.O_RDWR | os.O_CREAT, 0o600)
    loop = asyncio.get_running_loop()
    try:
        acquire = loop.run_in_executor(None, fcntl.flock, fd, fcntl.LOCK_EX)
        if timeout is None:
            await acquire
        else:
            try:
                await asyncio.wait_for(acquire, timeout=timeout)
            except asyncio.TimeoutError:
                raise GpuLockTimeout(f"gpu_lock timed out after {timeout}s") from None
        try:
            yield
        finally:
            fcntl.flock(fd, fcntl.LOCK_UN)
    finally:
        os.close(fd)
```

Notes:
- `loop.run_in_executor` with a blocking `flock` lets the event loop keep serving other coroutines while we wait.
- On `SIGKILL`, the kernel closes the fd and releases the lock — no stale-lock cleanup needed.
- On `asyncio.TimeoutError` we do NOT unlock (we never acquired); we DO close the fd (in the outer `finally`).
- `LOCK_EX` is blocking — the combination of `LOCK_EX + LOCK_NB` with a poll loop is also viable; the card chose the blocking-in-executor path for simplicity.

### D7 — Exception hierarchy placement
`merceka_core/errors.py`:
```python
class VideoUploadError(Exception): ...
class VideoBackendError(Exception): ...
class VideoNotFoundError(FileNotFoundError): ...
class GpuLockTimeout(TimeoutError): ...
```

Inheriting `VideoNotFoundError` from `FileNotFoundError` and `GpuLockTimeout` from `TimeoutError` means existing fallback cascades already catch them without registering the new types. That's the card's intent ("so the `fallback='gemma4:26b'` path fires on Anthropic 429/529/5xx").

### D8 — Mindweaver migration
- `mindweaver/backend/app/services/gpu.py`: replace the single line `GPU_LOCK = asyncio.Lock()` with `from merceka_core import gpu_lock as GPU_LOCK` — re-export the factory, not an instance.
- `mindweaver/backend/app/services/youtube.py:347`: change `lock_ctx = GPU_LOCK` to `lock_ctx = GPU_LOCK()`.
- `mindweaver/backend/app/services/youtube.py:476`: change `async with GPU_LOCK:` to `async with GPU_LOCK():`.
- Verify mindweaver's dependency on merceka_core is already present (it is, per `youtube.py:23` imports).
- Mindweaver's test suite (`uv run pytest`) must still pass.

The mindweaver PR lands **bundled in the same branch** as the merceka PR on the merceka side, because the card explicitly says "same PR that ships the merceka extension" and the mindweaver compile-time import would fail if the merceka side merges first and mindweaver hasn't caught up. But concretely: mindweaver is a separate repo — we'll open a mindweaver PR and a merceka PR, merge merceka first (mindweaver references merceka_core by installed path so there's no import error as long as the merceka side still exports the `asyncio.Lock()` attribute during the transition). The cleanest approach: land merceka with a NEW `gpu_lock` name but DO NOT delete anything merceka currently exports; then mindweaver switches imports in a follow-up commit on its own branch. That's two atomic PRs.

**Revised commit strategy:**
1. merceka_core PR: adds `gpu_lock`, exceptions, `generate_with_video`, OpenRouter-vision audit. No removal of anything. Safe-forward.
2. mindweaver PR: switches `GPU_LOCK` to `merceka_core.gpu_lock`. Depends on merceka PR being merged first (pip install from git main).
3. Run both test suites before merging either.

### D10 — Retry policy
Location: inside `_openrouter_call` / `_aopenrouter_call`. Codes: `408, 425, 429, 500, 502, 503, 504, 529`. Also `ConnectionRefusedError`, `ConnectionResetError`. Base 1s, factor 2, max 30s, max 3 attempts, +uniform [0, 1s) jitter. Respect `Retry-After` header when present.

4xx (except 408/425/429) never retry. This is a subtle behavior change — existing callers that get a 400 bad-request will now see it fail faster (before the fallback cascade). That's correct.

## Non-goals / out of scope

- Changing how Claude CLI (`model_name="claude/..."`) works. That remains text-stdin-only.
- Adding `anthropic` or `google-generativeai` as deps. OpenRouter covers Claude; `google-genai>=1.0` covers Gemini.
- A `gpu_lock` semaphore mode (Ollama NUM_PARALLEL > 1) — deferred per the card's follow-up list.
- Updating any slab/videotoframes/videototext/quest repo.

## Acceptance bar

A single `uv run pytest merceka_core/tests/ -m "not integration"` is green in default CI. The integration suite (`-m integration`) requires `GOOGLE_API_KEY` set and is a self-serve "I want to ship" gate. The cross-process GPU-lock integration test runs WITHOUT credentials; it's the `@pytest.mark.gpu` lock-specific test. Mindweaver's test suite (`cd mindweaver/backend && uv run pytest`) stays green after its migration commit.

## Rollback plan

If any capability turns out broken in production:
- `gpu_lock` is additive — callers can revert to `asyncio.Lock()` locally; the file lock just goes unused.
- Gemini video is a new method; not calling it is equivalent to not having it.
- OpenRouter vision currently works for Claude via `openrouter/anthropic/...` — our changes are in the exception-catching layer, so worst case a retry doesn't fire.
- Mindweaver migration reverts by flipping two import lines back.
